# Finance Dashboard ‚Äî Security and Quality Audit

This is a Next.js 16 / React 19 personal finance dashboard backed by Supabase (auth + RLS + RPCs) and Drizzle ORM against a direct Postgres connection. The app covers accounts, transactions, investments, mutual funds, bonds, forex, FnO, budgets, goals, liabilities, and family transfers. The most serious issues are a gating bypass on the sync API (fail-open when `CRON_SECRET` is absent), client-side-only login rate limiting, a browser Supabase singleton that misbehaves across tabs, and several logic bugs in dashboard trend computation.

**Watch for:** The `/api/sync` endpoint allows unauthenticated access in production when `CRON_SECRET` is unset, and it bypasses Supabase RLS via a raw Drizzle connection ‚Äî meaning an unauthenticated caller can trigger balance mutations for any user. The login lockout is trivially bypassable by calling the server action directly. The browser Supabase singleton self-destructs on sign-out in a way that creates a cross-tab race condition. The net-worth backward walk double-counts the current month's delta, producing incorrect historical chart data.

---

## High-level view

The middleware (`src/proxy.ts`) applies a strong CSP and gates all non-public routes, but `/api/sync` is listed in `PUBLIC_ROUTES` ‚Äî so the middleware skips auth for it entirely, and the route's own auth gate is the only line of defense. That gate has a fail-open bug: when `CRON_SECRET` is absent from the environment, the guard's `&&` short-circuits and the 401 is never returned. The route also uses a raw `pg.Pool` Drizzle connection that bypasses RLS, so a successful unauthenticated request can write balance mutations across all users.

The browser Supabase client nulls the module-level singleton on `SIGNED_OUT`. Any other open tab that subsequently calls `createClient()` gets a fresh unauthenticated client stored back as the singleton, while tabs that already captured the pre-null client reference continue using a stale object. Removing the null assignment entirely is safer ‚Äî the Supabase client manages session state internally.

Login rate limiting lives entirely in `localStorage` and a React ref. The `login` server action is a plain HTTP POST endpoint and can be called by any script, bypassing the browser entirely. Supabase Auth's built-in brute-force protection at the project level should be the actual mitigation.

The six-month net-worth trend in `DashboardClient` walks backward from the current `netWorthINR`. The loop starts at the current month's index and subtracts `income - expense` before moving to the previous month, which double-applies the current month's delta. Additionally, the backward investment estimate uses a hardcoded `income * 0.3 - expense * 0.1` heuristic that can produce negative values when expenses are high.

`updateInvestment` and `updateMFHolding` pass every optional field to Supabase's `.update()` without filtering undefined values. The Supabase JS client serializes `undefined` as `null` in the PATCH body, so any omitted field overwrites the database column with NULL. The same pattern is present in `updateBond` and `updateAlternativeAsset`.

The `checkApiHealth` server action in `settings/actions.ts` has no auth check, making it callable by any unauthenticated request. It fires eight outbound HTTP requests to external financial APIs ‚Äî a cost vector and an information leak about the app's external dependencies.

1. **`/api/sync` auth bypass when `CRON_SECRET` unset** ‚Äî `cronSecret && authHeader !== ...` short-circuits when `CRON_SECRET` is absent, skipping the 401. Flip the guard to deny by default: `if (!cronSecret || authHeader !== \`Bearer ${cronSecret}\`)`. [confirmed]

2. **`/api/sync` bypasses RLS via Drizzle** ‚Äî The sync route uses a raw `pg.Pool` Drizzle connection that runs with full Postgres credentials, bypassing all RLS policies. Recurring expense mutations execute for all users. Move the logic to a Supabase RPC or enforce per-user filtering explicitly in the loop. [confirmed]

3. **`/api/sync` in `PUBLIC_ROUTES` ‚Äî middleware never checks auth** ‚Äî Remove it from `PUBLIC_ROUTES` in `src/proxy.ts`; the middleware's session check should still run. [confirmed]

4. **Client-side-only login rate limiting** ‚Äî Lockout state lives in `localStorage` and a React ref. Calling the `login` server action directly, using incognito, or clearing storage bypasses it entirely. Rely on Supabase Auth's server-side brute-force protection instead. [confirmed]

5. **Browser singleton null-then-reinitialize race across tabs** ‚Äî On `SIGNED_OUT`, `supabase-browser.ts` sets `client = null`. Another tab's subsequent `createClient()` call stores a fresh unauthenticated client as the singleton. Remove the null assignment; let the Supabase client manage session state internally. [confirmed]

6. **`onAuthStateChange` subscription closure timing** ‚Äî `subscription.unsubscribe()` is called inside the callback that is defined before `subscription` is assigned. This works because the callback fires asynchronously today, but is fragile. Remove the self-unsubscribing pattern or assign the subscription to a module-level variable outside the callback. [confirmed]

7. **Net-worth backward walk double-counts current month** ‚Äî In `DashboardClient.tsx`, the loop starts at `monthsKeys.length - 1` (current month) and immediately subtracts that month's delta before moving backward. `netWorthINR` already incorporates this month's data, so the current month is double-applied. Skip the current month in the backward walk (`i = monthsKeys.length - 2`) or subtract the current delta before the loop. [confirmed]

8. **`runningInvestments` backward heuristic produces negatives** ‚Äî `runningInvestments -= (entry.income * 0.3 - entry.expense * 0.1)` in the same backward walk is an undocumented heuristic that goes negative when expenses are large. [confirmed]

9. **`checkApiHealth` requires no authentication** ‚Äî Fires eight outbound HTTP requests and a Supabase query with no `getUser()` check. Add an auth guard at the top of the function. [confirmed]

10. **`updatePassword` min-length is 6 characters** ‚Äî `reset-password/actions.ts` enforces 6 characters minimum. Raise to at least 8 for a finance application. [confirmed]

11. **`expenseTrend` ordering assumption** ‚Äî In `DashboardClient.tsx`, the first 15 expense transactions encountered in the unsorted `transactions` array are collected as "recent". If the RPC returns transactions in ascending order, these are the 15 oldest, not most recent. Sort `transactions` by `date` descending before the single-pass loop. [likely]

12. **`updateInvestment` / `updateMFHolding` / `updateBond` / `updateAlternativeAsset` ‚Äî undefined overwrites to NULL** ‚Äî All optional fields are spread directly into `.update(...)`. Supabase serializes `undefined` as `null`, overwriting existing column values. Build update payloads selectively, including only keys that are explicitly defined. [confirmed]

### `/api/sync` auth gate bypass and RLS escape

The auth check in `src/app/api/sync/route.ts`:

```typescript
if (process.env.NODE_ENV !== "development" && cronSecret && authHeader !== `Bearer ${cronSecret}`) {
  return NextResponse.json({ error: "Unauthorized" }, { status: 401 });
}
```

When `CRON_SECRET` is absent, `cronSecret` is `undefined`, the `&&` short-circuits, and the `return` never executes. The gate is fail-open. The correct form:

```typescript
const cronSecret = process.env.CRON_SECRET;
if (!cronSecret || authHeader !== `Bearer ${cronSecret}`) {
  return NextResponse.json({ error: "Unauthorized" }, { status: 401 });
}
```

This also removes the `NODE_ENV !== "development"` carve-out, which lets the endpoint run completely unprotected in dev ‚Äî and dev databases often contain real data.

The route uses `getDb()`, which opens a `pg.Pool` against `DATABASE_URL`. This credential is typically the Supabase `postgres` superuser or the migration user, bypassing all RLS policies. The recurring expense loop fetches **all** `is_recurring = true` expenses across all users and writes to `accounts`, `ledger_logs`, `transactions`, and `expenses`. There is no per-user ownership check after fetching the expenses ‚Äî `exp.user_id` is used in insert values but is never validated against a caller identity. An unauthenticated attacker who hits this endpoint can trigger balance deductions for every user in the database who has recurring expenses.

The route is also in `PUBLIC_ROUTES` in `src/proxy.ts`, so the middleware's Supabase session check never runs against it.

### Client-side login rate limiting

`failCountRef` is a React ref (reset on component unmount) and `localStorage` is per-origin, per-device. The `login` server action at `src/app/login/actions.ts` is a `"use server"` function ‚Äî it compiles to a POST endpoint at `/_next/action`. Any HTTP client can call it directly:

```bash
curl -X POST /_next/action -H "Next-Action: <hash>" -d '{"0":{"email":"...","password":"..."}}'
```

The 15-second lockout with `Math.pow(2, failCount - MAX_ATTEMPTS)` multiplier reaches 30s after 4 failures, 60s after 5 ‚Äî not a meaningful deterrent. Supabase Auth has server-enforced rate limiting on `signInWithPassword` configurable in the project dashboard. That is the appropriate control. The client-side lockout can remain as a UX affordance, but should never be framed as a security control.

### Browser singleton lifecycle and multi-tab behavior

`src/lib/supabase-browser.ts` sets `client = null` on `SIGNED_OUT`. The sequence in a two-tab scenario:

1. Tab A signs out ‚Üí `client = null`, subscription unsubscribed
2. Tab B had captured `client` before the null (via `UserContext`'s `useState(() => createClient())`) ‚Äî that reference still works and returns signed-out state from `getUser()` correctly
3. A new component mounts in Tab B and calls `createClient()` ‚Üí `client` is `null`, so a fresh unauthenticated client is created and stored as the singleton
4. Subsequent `createClient()` calls in Tab B now return an unauthenticated client

Removing the sign-out listener and the `client = null` assignment eliminates the race. The existing client object, even after sign-out, returns an empty session from `getUser()` ‚Äî no data leaks from the old session. The singleton's only purpose is to avoid multiple WebSocket connections; nulling it doesn't improve that.

### Dashboard net-worth trend backward walk

```typescript
// DashboardClient.tsx
let runningNetWorth = netWorthINR || 100000;  // current live value
for (let i = monthsKeys.length - 1; i >= 0; i--) {
  const entry = trendMap[monthsKeys[i]] as any;
  entry.netWorth = Math.max(0, runningNetWorth);  // assigns current value to current month
  runningNetWorth -= (entry.income - entry.expense); // then subtracts current month to go backward
}
```

The current month (index `monthsKeys.length - 1`) gets `netWorthINR` assigned correctly. But `runningNetWorth` is then decremented by the current month's net income, so the second-to-last month gets `netWorthINR - currentMonthDelta`. Since `netWorthINR` was computed from all transactions including this month, the backward step subtracts what was already in the base figure ‚Äî the current month is applied twice. The fix: start the loop at `i = monthsKeys.length - 2`, assign `netWorthINR` to the current month before the loop, then use the loop only for prior months.

The investment backward formula `runningInvestments -= (entry.income * 0.3 - entry.expense * 0.1)` has no documented basis and goes negative when `expense * 0.1 > income * 0.3` (i.e., when spending is more than 3√ó income). `Math.max(0, ...)` masks this, but the resulting chart values are meaningless.

### `updateInvestment` / `updateMFHolding` undefined-to-null overwrite

```typescript
// stocks/actions.ts ‚Äî updateInvestment
await supabase.from("investments").update({ 
  name: data.name,           // undefined if not passed ‚Üí serialized as null
  symbol: data.symbol,       // same
  quantity: data.quantity,   // same
  buy_price: data.buy_price, // same
  ...
}).eq("id", id).eq("user_id", user.id);
```

Callers that pass only `{ current_price: 145.5 }` will null out `name`, `symbol`, `quantity`, `buy_price`, `currency`, `notes`, and `bought_at`. The same pattern is in `updateMFHolding`, `updateBond` (which spreads all 20+ fields), and `updateAlternativeAsset`. Selective payload construction is the fix for all four.

### `checkApiHealth` ‚Äî unauthenticated outbound requests

The function has no `getUser()` call. Server actions compile to POST endpoints that Next.js serves without any inherent auth enforcement ‚Äî the auth check is the developer's responsibility. An unauthenticated request to this action:

- Reveals which financial data providers the app depends on and their current reachability
- Reveals the server's egress IP by probing external APIs
- Can be called in a loop to generate outbound traffic (rate-limited by the 8-second `AbortSignal.timeout`, but still)

### `expenseTrend` ordering assumption

```typescript
// DashboardClient.tsx
if (isRealExpense && expenseTrend.length < 15) {
  expenseTrend.push({ date: t.date, amount: tAmount, ... });
}
// ...later:
expenseTrend: expenseTrend.reverse()
```

`transactions` is the raw array from the SWR cache. The `get_summary_v1` RPC's ordering is not guaranteed on the client. If it returns oldest-first (typical Postgres default without an explicit ORDER BY), this collects the 15 oldest expenses, reverses them, and presents them as "recent". Sort `transactions` by `t.date` descending before the loop.

### Not tested

No tests exist for: any server action, `useFinanceData`, `useNetWorth`, the `login` flow, the `/api/sync` auth gate, or the middleware redirect logic. The auth gate bypass and the undefined-to-null overwrite are both regressions that a unit test would have caught. The `useNetWorth` backward walk bug requires a test with known income/expense data across months to be verified.
